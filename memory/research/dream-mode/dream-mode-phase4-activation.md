# Dream Mode Phase 4 Technical Spec: Knowledge Activation

---

## 1. Overview
Phase 4 activates eight capabilities: six existing underutilized runtime capabilities plus two new runtime consumers generated by Dream Mode analysis. It is **not** orchestrated by `DreamEngine`. Dream Mode Phases 1-3 collect, analyze, and consolidate the artifacts; Phase 4 is a separate set of `CoreLoop` and context-builder modifications that consume those outputs during normal execution.
This work touches three execution surfaces:
- `CoreLoop` decides when to retrieve and apply learned knowledge
- `context-builder` decides what knowledge enters the prompt
- `TaskLifecycle` and `StrategyManager` apply learned patterns during task and strategy generation

This is the point where SOAR/ACT-R-inspired consolidation becomes operational. Dream Mode does not only summarize past runs; it changes future action selection by adding priors, reusable procedures, and retrieval paths into the live system.
Design constraints:
- All activations are gated behind independent feature flags
- Default state is OFF for every capability
- Integrations must degrade safely to current behavior when dependencies are unavailable
- Each capability should be measurable in isolation

## 2. Capability 1: `selectForWorkingMemorySemantic`
### Current State
`selectForWorkingMemory()` is tag-based and already called in CoreLoop Phase B. A semantic variant, `selectForWorkingMemorySemantic()`, exists in [memory-selection.ts](/Users/yuyoshimuta/Documents/dev/SeedPulse/src/platform/knowledge/memory/memory-selection.ts#L439) but currently has no caller.

### Activation
In `src/orchestrator/execution/context/context-builder.ts`, after tag-based working-memory selection completes:

1. Check `dream.activation.semanticWorkingMemory`
2. If enabled, call `selectForWorkingMemorySemantic(currentTaskDescription, budget)`
3. Merge tag and semantic results with set union
4. Deduplicate by memory ID
5. Re-apply the existing budget cap

This preserves existing behavior while allowing semantic recall to recover relevant items that share meaning but not tags.

### Feature Flag
- `dream.activation.semanticWorkingMemory`
- Default: `false`

### Code Sketch
```ts
const tagSelected = await selectForWorkingMemory(task, budget);

let workingMemory = tagSelected;

if (config.dream.activation.semanticWorkingMemory) {
  const semanticSelected = await selectForWorkingMemorySemantic(
    task.description,
    budget
  );

  workingMemory = dedupeById([...tagSelected, ...semanticSelected]).slice(0, budget);
}
```

## 3. Capability 2: `searchCrossGoalLessons`
### Current State
`searchCrossGoalLessons()` exists in [memory-selection.ts](/Users/yuyoshimuta/Documents/dev/SeedPulse/src/platform/knowledge/memory/memory-selection.ts#L355) but is never called from `CoreLoop`.

### Activation
In `src/orchestrator/loop/core-loop.ts`, after task generation and before execution:

1. Check `dream.activation.crossGoalLessons`
2. Query `searchCrossGoalLessons()` using the active goal, current gap, and generated task description
3. Inject returned lessons into a dedicated execution-context slot such as `crossGoalLessons`

This allows the system to transfer lessons across goals without polluting the main memory-selection path.

### Feature Flag
- `dream.activation.crossGoalLessons`
- Default: `false`

---

## 4. Capability 3: KnowledgeGraph Traversal
### Current State
`KnowledgeGraph` stores typed edges such as `supports`, `contradicts`, `refines`, and `depends_on`, but those relations are not used when building runtime context.

### Activation
In `src/orchestrator/execution/context/context-builder.ts`, after the primary knowledge entries are selected:

1. Check `dream.activation.graphTraversal`
2. Traverse the graph one hop from each selected entry
3. Include `supports` and `refines` neighbors as additional context
4. Surface `contradicts` neighbors separately as warnings
5. Exclude or defer `depends_on` until a later phase unless explicitly needed for execution planning

This keeps traversal conservative and interpretable. The first goal is to improve recall, not to construct deep graph expansions.

### Feature Flag
- `dream.activation.graphTraversal`
- Default: `false`

---

## 5. Capability 4: `acquireWithTools`
### Current State
`KnowledgeManager.acquireWithTools()` exists but is never triggered by the live loop.

### Activation
In `src/orchestrator/loop/core-loop.ts`, when both of the following are true:

- `detectKnowledgeGap()` returns a positive result
- the loop is stalled or has crossed a configured stall threshold

Then:
1. Check `dream.activation.autoAcquireKnowledge`
2. Call `KnowledgeManager.acquireWithTools()` as a fallback path
3. Store acquired knowledge through the existing persistence path
4. Re-enter the loop with refreshed context instead of forcing immediate execution

This should be treated as the highest-risk activation because it adds external acquisition behavior to the control loop.

### Feature Flag
- `dream.activation.autoAcquireKnowledge`
- Default: `false`

---

## 6. Capability 5: `injectSemanticKnowledgeContext`
### Current State
`context-builder.ts` already contains both tag-based retrieval and a semantic two-pass progressive-disclosure path. Only the tag-based path is used today.

### Activation
Semantic retrieval becomes the preferred path when embeddings are available:
1. Check `dream.activation.semanticContext`
2. If enabled and an embedding client is configured, use `injectSemanticKnowledgeContext()`
3. If embeddings are unavailable or retrieval fails, fall back to the current tag-based path
4. Preserve token budgets and progressive-disclosure limits from the current implementation

This capability has the best value-to-risk ratio because the retrieval logic already exists and the fallback path is straightforward.

### Feature Flag
- `dream.activation.semanticContext`
- Default: `false`

---

## 7. Capability 6: LearnedPattern Retrieval at Task Generation
### Current State
`LearnedPattern` objects are stored with `embedding_id`, but they are not retrieved during task generation.

### Activation
In `src/orchestrator/execution/task/task-lifecycle.ts`, inside `TaskLifecycle.generateTask()`:

1. Check `dream.activation.learnedPatternHints`
2. Build a similarity query from the current gap description, failed attempts, and goal context
3. Query `VectorIndex` for nearby `LearnedPattern` entries
4. Inject the highest-confidence patterns into task-generation prompt inputs as hints, not hard constraints

This gives the task generator access to prior successful behaviors without forcing deterministic reuse.

### Feature Flag
- `dream.activation.learnedPatternHints`
- Default: `false`

---

## 8. Capability 7: Procedural Consolidation Output
### Current State
This is a new Phase 4 runtime consumer for Dream Mode Phase 3 output. Phase 3 extracts successful strategy sequences and stores them as reusable templates.

### Activation
In `src/orchestrator/strategy/strategy-manager.ts`, before generating free-form strategy candidates:

1. Check `dream.activation.strategyTemplates`
2. Query the strategy template library using current goal type, gap shape, and relevant context
3. If a high-confidence match exists, propose the template as a pre-built candidate
4. Evaluate it alongside normal generated candidates instead of bypassing the existing strategy-selection process

This is Voyager-inspired procedural consolidation: successful trajectories become reusable plans that can be replayed with adaptation.

### Feature Flag
- `dream.activation.strategyTemplates`
- Default: `false`

---

## 9. Capability 8: Decision Heuristics
### Current State
This is another new runtime consumer for Dream Mode Phase 3. Dream Mode learns compact decision rules such as "if stall count exceeds N under strategy X, pivot to strategy Y."

### Activation
In `src/orchestrator/strategy/strategy-manager.ts`, during candidate evaluation:

1. Check `dream.activation.decisionHeuristics`
2. Load matching heuristics for the current loop state
3. Apply them as lightweight priors, score nudges, or tie-breakers
4. Keep final selection under the existing strategy-evaluation framework

These heuristics do not replace LLM reasoning. They bias search toward historically effective actions, similar to SOAR-style learned productions.

### Feature Flag
- `dream.activation.decisionHeuristics`
- Default: `false`

## 10. Rollout Strategy
All capabilities default to OFF. They should be enabled one at a time so each can be measured independently and rolled back without affecting the others.
Operational guidance:
- Add a separate config flag for each activation
- Ship per-capability unit tests before enabling any flag in non-local environments
- Compare gap convergence speed, stall frequency, and recovery rate with and without each capability
- Watch for prompt bloat, irrelevant retrieval, and over-aggressive fallback acquisition

Recommended rollout order:
1. `semanticContext`
2. `crossGoalLessons`
3. `learnedPatternHints`
4. `graphTraversal`
5. `semanticWorkingMemory`
6. `strategyTemplates`
7. `decisionHeuristics`
8. `autoAcquireKnowledge`

This order starts with retrieval improvements, then adds procedural reuse, and leaves the highest-autonomy behavior for last.

## 12. Testing
### Unit Tests
Add focused tests for each capability:
- semantic working memory merges tag and semantic results, deduplicates, and respects budget
- cross-goal lessons are retrieved only when flag-enabled
- graph traversal includes `supports` and `refines`, and surfaces `contradicts` as warnings
- auto-acquire triggers only on knowledge gap plus stall
- semantic context uses embeddings when available and falls back safely
- learned pattern retrieval injects hints into task generation
- strategy templates appear as strategy candidates when matched
- decision heuristics adjust candidate evaluation without bypassing it

### Integration Test
Enable all flags in a test configuration and run `CoreLoop` against a seeded goal. Verify that the assembled runtime context includes:
- semantic knowledge results
- cross-goal lessons
- graph neighbors and contradiction warnings
- learned pattern hints
- strategy template candidates
- decision heuristic priors

The success condition is not only retrieval. The loop must continue to function correctly when every activation path is enabled simultaneously.
