# 知識獲得設計

> Conatusはゴール達成に必要な知識を最初から持っていない。しかし、自ら調べ、学び、理解する。
> 本ドキュメントは、未知ドメインの知識を能動的に調査・獲得・保存・活用するメカニズムを定義する。

> 関連: `observation.md`, `task-lifecycle.md`, `execution-boundary.md`, `session-and-context.md`, `curiosity.md`, `stall-detection.md`

---

## 1. 知識獲得の位置づけ

### コアループにおける役割

知識獲得はタスク発見ループ（mechanism.md 2）の各ステップに横断的に関わるが、主に2つのタイミングで発動する。

```
観測 → ギャップ認識 → 戦略選択 → タスク具体化 → (観測に戻る)
         ↑                ↑
     知識不足の検知     知識不足の検知
         │                │
         ↓                ↓
     調査タスク生成     調査タスク生成
```

**タイミング1: ギャップ認識時**
観測結果を解釈する段階で、「この観測値が意味するものがわからない」「この次元の正常値が不明」という状態を検知する。解釈に必要なドメイン知識が不足しているサインだ。

**タイミング2: 戦略選択時**
ギャップを埋める戦略を生成する段階で、「この領域の有効なアプローチが不明」「比較すべきベースラインがない」という状態を検知する。打ち手を考えるための知識が不足しているサインだ。

### 知識獲得の性質

知識獲得はゴール達成のための手段であり、それ自体が目的ではない。Conatusは「学ぶために学ぶ」のではなく「ゴールに近づくために必要な知識を調達する」。この区別は重要だ。知識獲得は常にゴールツリーのどこかのノードに紐づいていなければならない。

## 2. 知識不足の検知

知識不足は5種類のシグナルで検知される。いずれもコアループの中で自然に発生するものだ。

### 2.1 観測結果の解釈困難

```
シグナル: 観測値が得られたが、それが「良い」のか「悪い」のかを判断できない
発生場所: ギャップ認識ステップ
```

例: 犬の呼吸数が1分あたり28回と観測された。しかし、この犬種・年齢・健康状態における正常範囲がわからない。28回が危険なのか正常なのかを判断するためのドメイン知識がない。

**検知ロジック**: LLMによるギャップ認識の際に、「この値の評価に必要な基準知識が不足している」と判定された場合。判定結果に `knowledge_gap: true` フラグと `missing_knowledge: string` フィールドを付与する。

### 2.2 戦略生成の行き詰まり

```
シグナル: ギャップは明確だが、有効な戦略仮説を生成できない
発生場所: 戦略選択ステップ
```

例: SaaSのチャーン率が高いことはわかっている。しかし、チャーン改善の一般的な手法（コホート分析、リテンション施策の種類、業界ベンチマーク）についてのドメイン知識がなく、具体的な仮説を生成できない。

**検知ロジック**: 戦略選択LLM呼び出しが返す仮説の数がゼロ、または全仮説の確信度が極めて低い（0.3未満）場合。

### 2.3 停滞診断における情報不足判定

```
シグナル: 停滞検知後の原因分類で「情報不足」と診断された
発生場所: stall-detection.md 3.1
```

停滞が検知され、その原因が「観測の信頼度が低い次元で停滞が発生している」と分類された場合、知識不足がその根本原因である可能性がある。`stall-detection.md` 3.1 の「調査・検証タスクを生成する」対応の中に、知識獲得タスクが含まれる。

### 2.4 ゴール交渉時の新規ドメイン検知

```
シグナル: ゴールのドメインが、Conatusの経験ログに存在しない新領域である
発生場所: ゴール交渉ステップ（mechanism.md 3）
```

ユーザーが新しいゴールを設定したとき、そのドメインが経験ログに存在しなければ、次元分解・ベースライン設定・実現可能性評価のいずれにおいてもドメイン知識が必要になる。`goal-negotiation.md` の「要注意フラグ（新規ドメインや評価信頼度が低い場合）」が、知識獲得の初期トリガーとなる。

### 2.5 タスク生成時の前提知識不足

```
シグナル: タスクを具体化する際に、実行に必要な前提知識が明確でない
発生場所: タスク具体化ステップ
```

例: 「犬の呼吸器疾患のケアプランを作成する」というタスクを生成しようとしたが、呼吸器疾患のステージ分類、各ステージで推奨されるケア内容、投薬の種類と副作用についての知識がないため、成功基準もスコープも定義できない。

**検知ロジック**: タスク具体化LLM呼び出しが、成功基準の定義に失敗した、またはスコープ境界を「不明」と返した場合。

## 3. 調査タスクの生成

知識不足が検知されたら、Conatusは通常のタスク発見ループの中で**調査タスク**を生成する。調査タスクは通常のタスクと同じ構造（`task-lifecycle.md` 2）を持つが、いくつかの点で異なる性質を持つ。

### 3.1 調査タスクの構造

```
KnowledgeAcquisitionTask extends Task {
  task_category: "knowledge_acquisition"   // 通常タスクとの識別
  knowledge_target: string                 // 何についての知識を求めるか
  knowledge_questions: string[]            // 具体的に答えるべき問い（3〜5個）
  knowledge_scope: string                  // 調査の範囲制限
  expected_output_format: string           // 期待される成果物の形式
}
```

### 3.2 通常タスクとの違い

| 属性 | 通常タスク | 調査タスク |
|------|-----------|-----------|
| 目的 | ギャップを縮める | ギャップを理解する |
| 成功基準 | 状態ベクトルの変化 | 具体的な問いへの回答 |
| 状態変化 | current_valueが変わる | confidence / 知識ベースが変わる |
| 可逆性 | タスクによる | 常に `reversible`（情報収集は可逆） |
| 見積もり所要時間 | タスクによる | 調査タスクのデフォルト閾値を適用（`stall-detection.md` 2.2: 4時間） |
| スコープ | 変更を伴う | 情報収集に限定。変更を伴わない |

### 3.3 成功基準の設計

調査タスクの成功基準は「曖昧な理解」ではなく「具体的な問いへの回答」で定義する。

```
悪い成功基準: 「犬の呼吸器疾患について理解する」
良い成功基準:
  questions:
    - 「この犬種における呼吸数の正常範囲は何か」
    - 「呼吸数が正常範囲を超えた場合の緊急度分類は何か」
    - 「各緊急度における推奨される初期対応は何か」
  criterion:
    description: "3つの問いすべてに対して、信頼できる情報源に基づく回答が得られる"
    verification_method: "回答に情報源（文献・ガイドライン等）の引用が含まれるか確認"
    is_blocking: true
```

### 3.4 スコープ制限

調査タスクは発散しやすい。「もっと調べたい」という方向に際限なく広がる。これを防ぐために、調査タスクには明示的な制約を設ける。

**問いの数の制限**: 1つの調査タスクあたり3〜5個の具体的な問い。これを超える場合はタスクを分割する。

**深さの制限**: 調査の目的は「次のタスク生成に必要な知識を得ること」であり、「網羅的な知識体系を構築すること」ではない。目的に直結しない方向への探索は `out_of_scope` で明示的に除外する。

**時間の制限**: 調査タスクの `estimated_duration` は通常タスクより長めに設定するが、上限は設ける。無期限の調査は許容しない。

## 4. 知識獲得の手段

知識獲得タスクの実行は、`execution-boundary.md` に従い、すべてエージェントに委譲される。Conatusが自ら調査を行うことはない。

### 4.1 委譲可能な調査手段

| 手段 | 説明 | 委譲先の例 | 信頼度の目安 |
|------|------|-----------|------------|
| Web検索 | 公開情報の検索と要約 | 外部エージェント（Web検索ツール付き）、専用リサーチエージェント | 中 |
| ドキュメント読解 | 既存の技術文書・論文・ガイドラインの読解 | LLMプロバイダ経由のエージェント | 中〜高（ソースによる） |
| データ分析 | 既存データからのパターン発見 | データ分析エージェント | 高（データの質による） |
| 専門家への質問生成 | 人間の専門家に聞くべき質問の構造化 | LLM（質問生成） → 人間（回答） | 高 |
| ベンチマーク調査 | 業界標準・比較対象の特定 | リサーチエージェント | 中 |

### 4.2 調査手段の選択基準

Conatusは調査タスク生成時に、以下の優先順位で手段を選択する。

```
1. 既存データの分析（最もコストが低く信頼度が高い）
   → Conatusがすでにアクセスできるデータソースに答えがないか

2. ドキュメント読解（外部に出ずに済む場合）
   → ユーザーが提供したドキュメント、既知のリソース

3. Web検索（広く浅い情報収集）
   → 公開情報から基本的なドメイン知識を獲得

4. 専門家への質問生成（最もコストが高いが信頼度も高い）
   → 上記で解決しない場合、人間に聞くべき質問を構造化して提示
```

手段の選択は `execution-boundary.md` 5 の Capability Registry を参照する。利用可能な手段がない場合、人間にエスカレーションする。

### 4.3 段階的な知識獲得

1回の調査で完全な知識を得ることを期待しない。コアループの自己補正性質を活かし、段階的に知識を深める。

```
ループ1: Web検索で基本概念を把握
  → 「犬の呼吸器疾患には短頭種気道症候群・気管虚脱・肺炎がある」

ループ2: 得られた分類に基づいてより具体的な調査
  → 「この犬種はフレンチブルドッグ。短頭種気道症候群のリスクが高い」

ループ3: 具体的なケア基準の調査
  → 「短頭種気道症候群の呼吸数モニタリング基準は...」
```

各ループの調査結果が次のループの問いを精緻化する。このフィードバック構造により、最初から精密な問いを立てられなくても、ループを重ねるごとに必要な知識に収束していく。

## 5. 獲得した知識の保存

獲得した知識は `session-and-context.md` 8 の記憶3階層に統合して保存される。

### 5.1 記憶階層との対応

| 記憶階層 | 知識の保存内容 | 寿命 |
|---------|-------------|------|
| 作業記憶 | 今のタスクに必要な知識の抜粋 | セッション終了まで |
| ゴール状態 | ゴール達成に関連するドメイン知識 | ゴールが存在する限り |
| 経験ログ | 「この調査で何がわかったか」の記録 | Conatusインスタンスの存在する限り |

### 5.2 ドメイン知識ファイル

ゴール状態の一部として、ドメイン知識を専用ファイルに保存する。

```
~/.conatus/goals/<goal_id>/domain_knowledge.json
```

```
DomainKnowledge {
  goal_id: string
  domain: string                          // ドメイン識別子（例: "canine_respiratory"）
  entries: KnowledgeEntry[]
  last_updated: timestamp
}

KnowledgeEntry {
  entry_id: string                        // UUID
  question: string                        // 元の問い
  answer: string                          // 得られた回答
  sources: Source[]                       // 情報源
  confidence: number                      // 0.0〜1.0
  acquired_at: timestamp                  // 獲得日時
  acquisition_task_id: string             // 獲得元の調査タスクID
  superseded_by: string | null            // より新しいエントリで上書きされた場合
  tags: string[]                          // 検索用タグ（例: ["呼吸数", "正常範囲", "フレンチブルドッグ"]）
}

Source {
  type: "web" | "document" | "data_analysis" | "expert" | "llm_inference"
  reference: string                       // URL、ドキュメント名、専門家の識別子等
  reliability: "high" | "medium" | "low"  // 情報源の信頼性
}
```

### 5.3 知識の参照パス

保存された知識は、コンテキスト選択アルゴリズム（`session-and-context.md` 4）を通じてセッションに渡される。

```
タスク生成時:
  コンテキスト選択
    → 対象次元に関連するドメイン知識エントリを検索
    → tags と dimension_name の照合で関連エントリを抽出
    → 優先度6（経験ログの関連抜粋）としてコンテキストに含める
```

### 5.4 MVP vs Phase 2

| 段階 | 保存方式 | 検索方式 |
|------|---------|---------|
| **MVP** | ゴール別のJSONファイル。`tags` フィールドによるキーワードマッチ | タグの完全一致でフィルタリング |
| **Phase 2** | ゴール横断のナレッジベース。意味的埋め込みによるベクトル検索 | 類似度ベースの検索。異なるゴール間での知識共有 |

MVPでは、知識はゴールごとに独立して保存・参照される。ゴールAで得た知識をゴールBで活用するには、好奇心メカニズム（`curiosity.md` 4.3 クロスゴール転移）を通じた明示的な転用提案が必要だ。Phase 2で意味的検索が導入されれば、暗黙的な知識共有が可能になる。

## 6. 知識の検証

獲得した知識は無条件に信頼しない。信頼度の評価と矛盾検知を行う。

### 6.1 信頼度の評価

知識エントリの `confidence` は、以下の要素から決定する。

| 要素 | 高信頼度に寄与 | 低信頼度に寄与 |
|------|-------------|-------------|
| 情報源の種類 | 専門家の回答、査読済み文献 | LLM推論のみ、単一のWeb記事 |
| 情報源の数 | 複数の独立した情報源が一致 | 単一情報源のみ |
| 情報の新鮮さ | 最新の情報（1年以内） | 古い情報（5年以上前） |
| ドメインの安定性 | 安定した分野（物理法則等） | 急速に変化する分野（技術トレンド等） |

```
信頼度の目安:
  expert + 複数一致           → 0.85〜0.95
  査読済み文献               → 0.75〜0.90
  複数のWeb情報源が一致       → 0.60〜0.80
  単一のWeb情報源             → 0.40〜0.60
  LLM推論のみ（情報源なし）   → 0.20〜0.40
```

### 6.2 矛盾検知

新しい知識エントリが追加されたとき、既存のエントリとの矛盾を検出する。

**検出方法**: 同一の `tags` を持つエントリ間で、`answer` の内容が相反するかをLLMで判定する。

**矛盾が検出された場合の処理**:

```
矛盾検出
  │
  ├─ 信頼度に差がある場合（差 >= 0.2）
  │    → 高信頼度のエントリを採用
  │    → 低信頼度のエントリに superseded_by を設定
  │
  ├─ 信頼度が同程度の場合（差 < 0.2）
  │    → 両方を保持し、矛盾フラグを付与
  │    → 矛盾解消のための追加調査タスクを生成
  │
  └─ 信頼度が両方とも低い場合（両方 < 0.5）
       → 両方を保持するが、どちらもタスク生成の根拠には使わない
       → 「この領域の知識は不確実」として扱う
```

### 6.3 知識の陳腐化

ドメイン知識は時間とともに陳腐化する可能性がある。以下のメカニズムで対処する。

**定期的な再検証**: ゴールレビューセッション（`observation.md` 2 Layer 2）の中で、古い知識エントリ（`acquired_at` が一定期間以上前）の再検証を検討する。再検証が必要と判断された場合、同じ問いに対する新しい調査タスクを生成する。

**再検証のトリガー**: 知識エントリの `acquired_at` からの経過時間がドメインの安定性に応じた閾値を超えた場合。

| ドメインの安定性 | 再検証の目安 |
|----------------|------------|
| 安定（医学的知識の基本等） | 12ヶ月 |
| 中程度（業界慣行等） | 6ヶ月 |
| 変動（技術トレンド等） | 3ヶ月 |

## 7. 知識の活用

獲得した知識がコアループの各ステップにどうフィードバックされるか。

### 7.1 タスク生成への反映

ドメイン知識はタスク具体化の品質を直接向上させる。

```
知識獲得前:
  タスク: 「犬の健康をモニタリングする」
  成功基準: 不明確
  スコープ: 広すぎる

知識獲得後:
  タスク: 「フレンチブルドッグの呼吸数を継続モニタリングし、
           1分あたり30回を超えた場合にアラートを設定する」
  成功基準: 「呼吸数センサーが稼働し、閾値超過時に通知が届く」
  スコープ: 呼吸数モニタリングに限定
```

獲得した知識がタスク生成時のコンテキストに含まれることで、LLMはより具体的で検証可能なタスクを生成できる。

### 7.2 戦略選択への反映

ドメイン知識は戦略仮説の質を向上させる。

```
知識獲得前:
  戦略仮説: 「チャーン率を下げる」（具体性なし）

知識獲得後:
  戦略仮説:
    1. 「オンボーディング完了率を改善する（業界ベンチマーク: 65%、現在: 40%）」
    2. 「NPS調査をDay 7に実施し、detractorに早期介入する」
    3. 「料金プランの年間契約割引を導入する（同規模SaaSの平均解約率低下: 15%）」
```

### 7.3 観測精度への反映

ドメイン知識は観測結果の解釈精度を向上させる。

```
知識獲得前:
  観測値: 呼吸数28回/分
  解釈: 不明（知識不足）
  confidence: 0.30

知識獲得後:
  観測値: 呼吸数28回/分
  解釈: 短頭種の安静時正常範囲（15-30回/分）内だが上限に近い。注意観察を推奨
  confidence: 0.80
```

### 7.4 ゴール交渉への反映

新しいゴールの設定時に、蓄積されたドメイン知識が実現可能性評価の精度を向上させる。

```
初回の犬の健康ゴール設定:
  実現可能性評価: 「新規ドメイン。評価信頼度が低い」 → 要注意フラグ

2回目の別の犬の健康ゴール設定:
  実現可能性評価: 蓄積されたドメイン知識を参照し、具体的な数値に基づく評価が可能
```

## 8. MVP vs Phase 2

### MVP

| 要素 | MVP実装 |
|------|--------|
| 知識不足の検知 | LLMの判定結果に `knowledge_gap` フラグを付与。2.1（解釈困難）と 2.2（戦略行き詰まり）を優先実装 |
| 調査タスクの生成 | `task_category: "knowledge_acquisition"` を持つ通常タスクとして生成。構造は 3.1 に準拠 |
| 調査手段 | Web検索（エージェントのツール経由）とドキュメント読解に限定。専門家への質問生成はエスカレーションとして扱う |
| 知識の保存 | ゴール別JSONファイル（`domain_knowledge.json`）。タグベースの完全一致検索 |
| 知識の検証 | 情報源の種類と数に基づく信頼度の自動設定。矛盾検知はLLMによる簡易判定 |
| 知識の活用 | タスク生成時のコンテキストに関連知識エントリを含める |
| 知識の陳腐化対処 | 未実装。手動での再調査に依存 |

## 設計原則のまとめ

| 原則 | 具体的な設計決定 |
|------|----------------|
| 知識獲得はゴール追求の一部 | 独立した学習モードではなく、コアループの中で自然に発動する |
| 具体的な問いで駆動 | 「調べる」ではなく「この3つの問いに答える」が成功基準 |
| 委譲モデルの遵守 | Conatusは調査を指示する。調査を実行するのはエージェント |
| 段階的な深化 | 1回で完全な知識を求めない。ループの反復で精緻化する |
| 知識は検証する | 獲得した知識の信頼度を評価し、矛盾を検知する |
| スコープの制御 | 調査の発散を防ぐ。問いの数と深さを制限する |
