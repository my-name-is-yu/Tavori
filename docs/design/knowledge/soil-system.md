# Soil System Design

> Soil is PulSeed's human-readable knowledge surface. Runtime JSON remains the write truth; Soil Markdown and its local index are derived read-through projections for agents and humans.

> Related: `memory-lifecycle.md`, `learning-pipeline.md`, `knowledge-acquisition.md`, `reporting.md`, `schedule-engine.md`

> Current implementation note: Soil is not just a documentation surface. `soil_query` is now available to the native AgentLoop and to selected CoreLoop phases such as `knowledge_refresh`, `replanning_options`, and `verification_evidence`, making Soil part of live execution rather than a passive export.

---

## 1. Contract

The Soil system has a strict source-of-truth boundary.

| Layer | Role | Mutability |
|-------|------|------------|
| Runtime JSON | Write truth for reports, schedules, knowledge, agent memory, and decisions | Mutated by runtime subsystems |
| Soil Markdown | Human-readable projection of runtime truth | Generated by PulSeed; humans may add marked overlays |
| `.index/soil.db` | Query index over Soil Markdown | Rebuilt from Soil Markdown |

Soil is the read-through surface. Runtime code writes JSON first, then projects a Markdown page into Soil. Retrieval reads Soil and its index. This keeps operational truth stable while making accumulated memory inspectable in editors such as Obsidian.

The file named `.index/soil.db` is currently a `file-json-v1` snapshot, not a native SQLite database. The path is intentionally stable so the storage engine can evolve later without changing callers.

---

## 2. Projected Content

Soil contains generated Markdown pages for the main long-running PulSeed memory surfaces.

| Runtime source | Soil projection |
|----------------|-----------------|
| Reports | Report pages under Soil report paths, including generated summaries and source watermarks |
| Schedules | Current schedule/task overview pages |
| Domain knowledge | Per-goal knowledge pages |
| Shared knowledge | Cross-goal shared knowledge pages |
| Agent memory | Agent memory pages derived from the runtime memory store |
| Decisions | Recent decision pages, including empty aggregate pages when no decisions remain |
| Identity | Mirrors for `SEED.md`, `ROOT.md`, and `USER.md` |
| System pages | Soil index/status/system overview pages |

Each generated page carries frontmatter describing its `soil_id`, source truth, source references, generation watermark, renderer, checksum, stale status, and manual overlay state. The checksum is used by doctor/index checks to detect drift between the page, its metadata, and the indexed snapshot.

---

## 3. Rebuild and Pruning

`soil_rebuild` regenerates Soil from runtime JSON and then rebuilds the index.

The rebuild process:

1. Reads runtime JSON from the configured base directory.
2. Reprojects reports, schedules, knowledge, agent memory, decisions, identity, and system pages.
3. Prunes generated runtime projections whose referenced runtime source files no longer exist.
4. Moves pruned pages into `.stale/pruned/` instead of deleting them outright.
5. Rebuilds `.index/soil.db` from the remaining Soil Markdown pages.

This prevents deleted runtime truth from continuing to appear as fresh queryable Soil content while preserving an audit trail for humans.

---

## 4. Retrieval

`soil_query` is the read path for agents.

It supports:

| Mode | Behavior |
|------|----------|
| Query text | Searches the Soil index when fresh |
| `soil_id` | Loads the matching Markdown page directly |
| Path | Loads the matching Markdown page directly |

For text search, `soil_query` prefers `.index/soil.db`. If the index is missing or stale relative to current Markdown, it falls back to a bounded manifest scan and returns a warning. This keeps reads useful after direct Markdown edits while still making index drift visible.

---

## 5. Manual Overlays

Humans are allowed to add marked overlay blocks to Soil Markdown:

```md
<!-- soil:overlay-begin -->
Human-authored note or correction.
<!-- soil:overlay-end -->
```

`soil_import` scans these blocks and stores candidates in the local overlay queue. It supports scan, list, approve, and reject actions.

Approval is bookkeeping only. The importer does not mutate runtime JSON truth. A future runtime import path can consume approved overlay candidates explicitly, but the implemented contract keeps manual edits from silently changing operational state.

---

## 6. Health Checks

`soil_doctor` inspects the Soil root and reports material consistency issues:

| Check | Purpose |
|-------|---------|
| Invalid frontmatter | Detect malformed Soil pages |
| Duplicate `soil_id` | Prevent ambiguous direct lookup |
| Checksum drift | Detect page/index/frontmatter mismatch |
| Watermark mismatch | Detect projection/source inconsistency |
| Missing source paths | Detect generated pages whose runtime sources are gone |
| Missing or stale index | Detect query index drift |

Doctor findings are diagnostic. They do not mutate Soil by themselves.

---

## 7. Tool Summary

| Tool | Permission class | Purpose |
|------|------------------|---------|
| `soil_query` | Read-only | Retrieve Soil pages or search Soil content |
| `soil_doctor` | Read-only | Inspect Soil health and index consistency |
| `soil_rebuild` | Local write with approval | Rebuild projections and index from runtime JSON |
| `soil_import` | Local write with approval | Maintain the manual overlay import queue |

The core invariant is: JSON writes are authoritative; Soil reads are ergonomic. Soil can be edited by humans, but direct edits become visible through overlays, stale-index warnings, and doctor findings rather than silently replacing runtime truth.
