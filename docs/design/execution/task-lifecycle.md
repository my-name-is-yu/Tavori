# Task Lifecycle Design

> Defines the full process from when a task is generated to when it is completed or fails.
> This document concretely describes "what a task is," "how it is created, executed, and concluded."

---

## 1. Separating Task Selection from Task Generation

One of the most important design decisions in the task discovery loop (`mechanism.md` §2.3–2.4) is the separation of "selection" from "generation."

### Why Separate Them?

Handing the entire question of "what should be done next" to an LLM creates two problems.

**Problem 1: Trying to do everything at once**
LLMs keep picking up "potentially relevant" things from context. The scope keeps expanding — "fix the tests, update the docs, rethink the design." Tasks become bloated and impossible to verify or recover from.

**Problem 2: Prioritizing the interesting over the important**
LLMs are pulled by their current context. They tend to drift toward "recently discussed issues" or "interesting problems to solve," missing the gaps that most directly advance the goal.

### Structure of Separation

```
[Code] Select which dimension to address
    ↓ Priority based on drive score (numerical calculation)
    ↓ Target dimension is determined

[LLM] Generate how to address that dimension
    ↓ Description of the concrete work to be done
    ↓ Articulation of success criteria and scope boundaries
```

"Which dimension to address" is determined mechanically by a number (drive score). The LLM's job is only to specify "how to address it." This lets PulSeed leverage the LLM's creativity while keeping authority over prioritization in code.

### Overview of Selection Logic

The code performs the following calculation:

1. Retrieve drive scores for all dimensions (sum of deadline-driven, dissatisfaction-driven, and opportunity-driven scores)
2. Extract the top N dimensions by score as candidates
3. Filter by uncertainty constraint (dimensions with low confidence are routed to observation tasks)
4. Further narrow by resource constraints (maximum M dimensions per iteration)
5. Pass the confirmed target dimensions to the LLM

## 2. Task Structure

A task consists of the following elements. Only when all elements are in place is a task considered "ready to execute."

### 2.1 Target Dimensions

```
target_dimensions: string[]   // identifiers of the dimensions to address (one or more)
primary_dimension: string     // the primary dimension (the priority axis when there are multiple)
```

Typically one dimension. Closely related dimensions (e.g., `test_coverage` and `stability`) may be targeted simultaneously. However, packing loosely related dimensions into the same task is prohibited.

### 2.2 Work Description

```
work_description: string      // what to do (generated by LLM)
rationale: string             // why this approach was chosen
approach: string              // the specific approach (outline of steps)
```

The part generated by the LLM. Rather than vague expressions ("improve," "optimize"), this describes concretely what will be changed and how.

### 2.3 Success Criteria

```
success_criteria: Criterion[]
```

Each criterion has the following:

```
Criterion {
  description: string         // what constitutes completion
  verification_method: string // how to verify (run tests, check file, etc.)
  is_blocking: boolean        // if false, task can complete even if this criterion is not met (desirable but not required)
}
```

Success criteria must be **verifiable**. "Quality improved" is not a criterion. "All tests pass and coverage is 80% or above" is a criterion.

### 2.4 Scope Boundary

```
scope_boundary {
  in_scope: string[]          // what this task touches
  out_of_scope: string[]      // what this task does not touch (explicit exclusions)
  blast_radius: string        // the range of potential impact if the task fails
}
```

Explicitly stating `out_of_scope` is important. Executors (including LLMs) face the temptation to "do this while I'm at it." Explicitly declaring what not to do structurally prevents scope creep.

### 2.5 Inherited Constraints

```
constraints: Constraint[]     // constraints inherited from the goal
```

Constraints defined at the goal level ("never write directly to the production DB," "require human approval when modifying user data," etc.) are passed down. The task does not need to know the goal's constraints — they are simply inherited and delivered.

### 2.6 Intentional Wait (plateau_until)

```
plateau_until: DateTime | null
```

When the strategy layer plans an intentional wait for a task or dimension, it sets the expected end datetime here. `null` means no wait (normal state).

| State | Meaning |
|-------|---------|
| `null` | No wait. Stall detection operates normally |
| `DateTime` (future) | Intentional wait in progress. Stall detection is suppressed until this datetime |
| `DateTime` (past) | Wait period has ended. Stall detection resumes |

**Usage example**: Used when intentionally pausing the loop to measure external changes, such as waiting 3 days for A/B test results or observing campaign effects for 1 week.

This field is set by the strategy layer (the LLM when generating tasks). Executors are prohibited from changing it independently. When `plateau_until` is set, the suppression logic in `stall-detection.md` §2.5 applies.

### 2.7 Estimated Duration

```
estimated_duration: Duration | null   // estimated time, as estimated by the LLM at task generation
```

Estimated by the LLM at task generation. Expressed in minutes or hours (e.g., `{ value: 3, unit: "hours" }`).

`null` is allowed. Set to `null` for tasks where estimation is not possible (exploratory research, tasks with many external dependencies).

Handling when `null` for stall detection (`stall-detection.md` §2.2): The time overrun check falls back to an absolute time threshold based on the goal's domain and past performance.

### 2.8 Consecutive Failure Count

```
consecutive_failure_count: number     // consecutive failure count for the same type of task (default: 0)
```

**This is the single source of truth for the failure counter in the entire system.** Both the consecutive failure detection in `stall-detection.md` and the repeated failure trigger condition in `curiosity.md` reference this field.

The unit of counting is the combination of "same dimension × same task category." The count resets when a different-category task is generated using a completely different strategy.

**Escalation threshold: 3 times** (configurable parameter: `failure_escalation_threshold`). When this threshold is exceeded, the `escalate` judgment in `task-lifecycle.md` §6 is triggered. To prevent the same number from being scattered across multiple documents, it is centrally managed here.

> **Note**: The stage counter for the graduated response (1st–3rd detection) in `stall-detection.md` §4 tracks the **escalation stage** of stall detection and is a separate concept from `consecutive_failure_count`. The former tracks "which detection occurrence this is," while the latter tracks "the consecutive failure count for the same type of task." Both share a threshold of 3, but their reset conditions, scope, and purposes are independent.

### 2.9 Reversibility Tag

```
reversibility: "reversible" | "irreversible" | "unknown"
```

The LLM determines and attaches the reversibility of the task at task generation time.

| Value | Meaning |
|-------|---------|
| `reversible` | Can be rolled back or undone if it fails |
| `irreversible` | Cannot be undone once executed (deletions, external sends, deployments, etc.) |
| `unknown` | Cannot be determined |

**`unknown` is treated the same as `irreversible` (conservative principle).** Optimistically treating the undetermined as "probably fine" is prohibited.

This tag is referenced by the approval gate in `trust-and-safety.md` §4 before execution begins (see `trust-and-safety.md` §4 for details). Tasks judged as `irreversible` or `unknown` always require human approval, regardless of trust balance or confidence score.

## 3. Scope Sizing

A task must be sized so that it "can be completed in a single execution session."

### Sizing Criteria

**Single focus**: The target is one dimension, or a group of closely related dimensions. "Frontend UI, backend API, and infrastructure settings all at the same time" is not a single focus.

**Verifiable success criteria**: After execution, completion can be objectively judged. Criteria that rely solely on subjective evaluation are a sign the scope is too large.

**Limited blast radius**: If the task fails, its impact does not propagate to other dimensions. The design ensures "other progress is protected even if this task fails."

### Sizing Failure Patterns

| Pattern | Symptom | Response |
|---------|---------|---------|
| Oversized | Success criteria contain 3 or more independent conditions | Split the task |
| Undersized | Success barely narrows the gap at all | Consider merging with related tasks |
| Vague | Judging completion requires further subjective judgment | Concretize the success criteria |

## 4. Execution

The task is handed off to the execution layer (agent, API, human, etc.). PulSeed does **not intervene** during execution.

### What PulSeed Passes

At execution start, a context is assembled containing the task's scope, success criteria, and constraints, and passed to the executor (see `runtime.md` §3). Information passed is limited to the "minimum necessary to complete this task." The full context of the goal is not needed.

### What PulSeed Monitors

```
execution_state {
  status: "running" | "completed" | "timed_out" | "error"
  started_at: timestamp
  timeout_at: timestamp
  heartbeat_at: timestamp      // time of last confirmed response
}
```

This is all PulSeed checks. It detects three types of state changes — "progress is not visible," "timeout is approaching," "abnormal termination" — and responds accordingly. It does not intervene in the execution itself.

### Non-Intervention During Execution Principle

This prevents contaminating the executor's context. Inserting "actually, fix this too" or "by the way, this issue as well" mid-execution collapses the scope and makes verification impossible. Instructions are finalized before execution begins.

## 5. Verification (3-Layer Structure)

After execution completes, PulSeed performs a 3-layer verification. This structure corresponds to the "result verification" described in `runtime.md` §1.

### Layer 1: Mechanical Verification

```
mechanical_checks: Check[]    // automated, objective checks
```

Test execution results, file existence checks, build success/failure, API response status. This has the highest confidence. It is the kind of evidence that cannot lie.

Mechanical verification is performed **in a separate process from the execution session**. Even if the executor reports "tests passed," PulSeed verifies this independently.

#### Layer 1 Execution Mechanism

Layer 1 is executed by PulSeed launching a dedicated **verification session** (completely separate from the execution session).

What the verification session receives:
- Task definition (success criteria + scope boundary)
- Expected deliverables (files, API endpoints, build artifacts, etc.)
- Access to deliverables produced by the execution session

Verification content by task type:

| Task type | What Layer 1 does |
|-----------|------------------|
| Code tasks | Run tests, type checks, lint (via agent session) |
| Non-code tasks | Check for expected files, validate format, check API response codes |

**Execution timing**: Immediately after the execution session completes, before Layer 2.

**When Layer 1 is not applicable**: When no mechanical verification method exists for the task type, Layer 1 is skipped. In this case, verification uses only Layer 2 + Layer 3, and the final confidence score is reduced before a judgment is made.

Confidence: **High**

### Layer 2: Task Reviewer

```
reviewer_session {
  input: {
    success_criteria,          // success criteria (no execution context)
    artifacts,                 // access to deliverables
    reference_standards        // evaluation standard information
  }
  output: {
    criteria_met: boolean[]    // fulfillment status of each success criterion
    quality_assessment: string // qualitative evaluation
    concerns: string[]         // concerns
    confidence: number         // 0.0–1.0
  }
}
```

An independent LLM session separate from execution evaluates the deliverables. The key is that **the execution session's context is not passed**. Precisely because the execution context is not shared, an unbiased judgment can be made.

Confidence: **Medium**

### Layer 3: Executor's Self-Report

```
executor_report {
  completed: boolean
  summary: string              // what was done
  partial_results: string[]    // completed portions (if incomplete)
  blockers: string[]           // reasons for failure
}
```

Treated as reference information. Self-report alone is not used to determine completion. However, "what could not be done" and "what was blocking" are used in failure response decisions.

Confidence: **Low**

### Conflict Resolution Rules Between Layer 1 and Layer 2

When Layer 1 (mechanical verification) and Layer 2 (task reviewer) return different judgments, the following rules resolve the conflict:

| Layer 1 | Layer 2 | Resolution |
|---------|---------|-----------|
| PASS | PASS | **PASS** (both agree) |
| FAIL | FAIL | **FAIL** (both agree) |
| PASS | FAIL | **Investigate**: Layer 2's concern may indicate a semantic issue (the kind that Layer 1 cannot detect). Re-run the Layer 2 review session with additional context. If still FAIL after re-run, treat as **FAIL**. |
| FAIL | PASS | **FAIL**: Layer 1 (mechanical, objective) takes precedence. Layer 1 failures cannot be overridden by subjective evaluation. |

**Rationale**: Layer 1 verifies objective facts such as "did the tests pass" and "does the file exist." These have clearer definitions than Layer 2's subjective evaluation of "is the quality sufficient." However, when Layer 1 passes but Layer 2 raises a concern, there is a possibility of a semantic defect despite mechanical passage (e.g., the types are right but the behavior is wrong), so additional review is performed to confirm.

### Integrating Verification Results

The results of the 3 layers are integrated to assign a confidence score to the task. This score is input to the subsequent gap recognition (`mechanism.md` §2.2).

```
verification_result {
  verdict: "pass" | "partial" | "fail"
  confidence: number           // integrated confidence
  evidence: Evidence[]         // observations that served as evidence
  dimension_updates: DimensionUpdate[]  // score changes for each dimension
}
```

## 6. Failure Response Decision Logic

When a task does not fully succeed, PulSeed selects a response from three options.

### Decision Flow

```
Task failed / partially completed
    │
    ├─ Is the direction correct?
    │       ↓ Yes (partial results are heading toward the goal)
    │       Can the results be reused?
    │             ↓ Yes → keep
    │             ↓ No  → don't retain, pivot direction → discard
    │
    ├─ Is the direction wrong?
    │       ↓ Yes (results are not heading toward the goal, or making things worse)
    │       → discard
    │
    └─ Has it failed repeatedly?
            ↓ Yes (same type of task has failed 3 or more times)
            → escalate
```

### keep: Retain Partial Results and Continue

Applicable conditions:
- Partial results are progressing in the goal direction
- The incomplete portion is clearly identifiable
- The remaining work can be defined as a separate task

Processing:
- Reflect the completed portion's results in state
- Generate the incomplete portion as a new task
- Prefix the new task with context indicating "remaining work from the previous task"

Example: 7 out of 10 features are implemented → Record the gap reduction for 7 features and pass the remaining 3 features to the next task.

### discard: Discard Results and Change Direction

Applicable conditions:
- The direction of the approach was wrong
- Leaving the results as-is would contaminate or impede other dimensions
- There is no prospect of improvement from continuing the same approach

Processing:
- Revert the changes if possible (see "Determining Revertibility" below)
- Pass a record of the failed approach to the next task generation context (to avoid repeating the same approach)
- Re-generate a task with a different approach

Example: A data structure change broke existing functionality → Revert the change and generate a task to investigate the affected scope first.

#### Determining Revertibility

When taking the discard path, whether to attempt a revert is determined by referencing the task's `reversibility` tag (§2.9). Also refer to `trust-and-safety.md` §4 for the definition of irreversible actions.

| `reversibility` value | Revert handling |
|----------------------|----------------|
| `reversible` | Launch a new agent session and attempt to revert |
| `irreversible` | Do not attempt a revert. Mark the dimension state as `dirty` (contaminated) and escalate to a human |
| `unknown` | Treat the same as `irreversible` (conservative principle) |

#### Handling Revert Failure

When a revert session is launched for a task judged `reversible` but the revert itself fails:

1. **Escalate to a human**: Notify with all of the following information.
   - Original task definition and execution results
   - Contents of the failed deliverables
   - Contents of the attempted revert and the reason for its failure

2. **Update dimension state**: Set the following flag on the target dimension.

```
state_integrity: "uncertain"   // manual observation required
```

For dimensions with `state_integrity: "uncertain"` set, PulSeed does not autonomously select and execute the next task. That dimension is paused until a human manually confirms and updates the state, and returns `state_integrity` to `"ok"`.

### escalate: Request Human Intervention

Applicable conditions:
- The same type of task has failed 3 or more times
- PulSeed does not have the necessary permissions or capabilities (production environment access, legal judgment, etc.)
- The premises may be wrong and the goal itself needs reconsideration

Processing:
- Record the escalation reason (what was tried and why it failed)
- Notify the human (explicitly state what is being requested)
- Enter a waiting state (shift attention to other goals while awaiting a response)

**Core of the decision**: "Did the work move in the goal direction?" If incomplete but direction was correct, keep. If direction was wrong, discard. If unable to resolve autonomously, escalate.

## 7. Task Completion → Continuing the Loop

When a task is completed (or ultimately processed), PulSeed immediately launches the next loop.

```
Task completed
    │
    ↓
Update state vector
  Update current values for each dimension based on verification results
  Also update observation confidence
    │
    ↓
Recalculate gap
  Recalculate gaps for all dimensions based on the updated state vector
    │
    ↓
Recalculate drive score
  Update deadline-driven, dissatisfaction-driven, and opportunity-driven scores
  (opportunity score also decays over time)
    │
    ↓
Select and generate next task
```

The loop never stops. Task completion is a trigger to begin the next task. This cycle continues until the goal is achieved or the user stops it.

---

## Summary of Design Principles

| Principle | Concrete design decision |
|-----------|------------------------|
| Code decides priority | Dimension selection by drive score is not delegated to the LLM |
| Separation of execution and verification | Verification is performed in a separate session and context from execution |
| Explicit scope boundaries | Explicitly stating `out_of_scope` structurally prevents "while I'm at it" scope creep |
| Failure is information | Failed approaches are recorded and passed to the next generation |
| The loop never stops | Task completion is not an endpoint — it is the starting point for the next loop |
