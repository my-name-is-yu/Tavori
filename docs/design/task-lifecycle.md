# タスクライフサイクル設計

> タスクが生成されてから完了または失敗に至るまでの全プロセスを定義する。
> 本ドキュメントは「タスクとは何か」「どう作られ、どう実行され、どう終わるか」を具体的に記述する。

---

## 1. タスク選択とタスク生成の分離

タスク発見ループ（mechanism.md §2.3–2.4）の中で、最も重要な設計判断の一つが「選択」と「生成」の分離だ。

### なぜ分離するのか

LLMに「次に何をすべきか」を丸ごと委ねると、2つの問題が生じる。

**問題1: 一度に全部やろうとする**
LLMは文脈の中で「関係しそうなこと」を拾い続ける。「テストも直して、ドキュメントも更新して、設計も見直して」という広がりが止まらない。タスクが肥大化し、検証も失敗対応も困難になる。

**問題2: 重要なものより興味深いものを優先する**
LLMは現在のコンテキストに引っ張られる。「直近で話題になった課題」や「解くのが面白い問題」に流れやすく、ゴール達成に最も効くギャップを外すことがある。

### 分離の構造

```
[コード] どの次元を攻めるか選択
    ↓ ドライブスコアに基づく優先順位（数値計算）
    ↓ 対象次元が確定する

[LLM] その次元をどう攻めるか生成
    ↓ 具体的な作業内容の記述
    ↓ 成功基準・スコープ境界の言語化
```

「何を攻めるか」は数値（ドライブスコア）で機械的に決まる。LLMの仕事は「どう攻めるか」の具体化だけだ。これにより、LLMの創造性を活かしながら、優先順位の決定権はコードに留める。

### 選択ロジックの概要

コードが行うのは以下の計算だ。

1. 全次元のドライブスコアを取得（締切駆動・不満駆動・機会駆動の合算）
2. スコア上位N次元を候補として抽出
3. 不確実性制約でフィルタリング（信頼度が低い次元は観測タスクへ）
4. リソース制約でさらに絞る（1イテレーションあたり最大M次元）
5. 確定した対象次元をLLMに渡す

## 2. タスク構造

タスクは以下の要素で構成される。すべての要素が揃って初めてタスクは「実行可能な状態」となる。

### 2.1 ターゲット次元

```
target_dimensions: string[]   // 攻める次元の識別子（1つ以上）
primary_dimension: string     // 主要な次元（複数の場合の優先軸）
```

通常は1次元。密接に関連する次元（例: `test_coverage` と `stability`）が同時対象になることがある。ただし、関連が薄い次元を同一タスクに詰め込むことは禁止する。

### 2.2 作業内容

```
work_description: string      // 何をするか（LLMが生成）
rationale: string             // なぜこの方法を選んだか
approach: string              // 具体的なアプローチ（手順の概要）
```

LLMが生成する部分。曖昧な表現（「改善する」「最適化する」）ではなく、何を・どのように変えるかを具体的に記述する。

### 2.3 成功基準

```
success_criteria: Criterion[]
```

各基準は以下を持つ。

```
Criterion {
  description: string         // 何をもって完了とするか
  verification_method: string // どう確認するか（テスト実行、ファイル確認、etc.）
  is_blocking: boolean        // falseなら未達でも完了可（望ましいが必須でない）
}
```

成功基準は**検証可能**でなければならない。「品質が上がった」は基準ではない。「全テストが通過し、カバレッジが80%以上」が基準だ。

### 2.4 スコープ境界

```
scope_boundary {
  in_scope: string[]          // このタスクが触ること
  out_of_scope: string[]      // このタスクが触らないこと（明示的な除外）
  blast_radius: string        // 失敗したときに影響しうる範囲
}
```

`out_of_scope` の明示が重要だ。実行者（LLMを含む）は「ついでにやろう」という誘惑にさらされる。何をしないかを明示することで、スコープ外への滑り込みを構造的に防ぐ。

### 2.5 継承された制約

```
constraints: Constraint[]     // ゴールから継承された制約
```

ゴールレベルで定義された制約（「本番DBには直接書き込まない」「ユーザーデータを変更する場合は人間承認を要求」など）を引き継ぐ。タスクはゴールの制約を知らなくてよい——継承されて渡されるだけだ。

### 2.6 意図的待機（plateau_until）

```
plateau_until: DateTime | null
```

戦略層がタスク・次元に対して意図的な待機を計画するとき、終了予定日時を設定する。`null` は待機なし（通常状態）を意味する。

| 状態 | 意味 |
|------|------|
| `null` | 待機なし。停滞検知は通常通り動作する |
| `DateTime`（未来） | 意図的な待機中。停滞検知をこの日時まで抑制する |
| `DateTime`（過去） | 待機期間が終了。停滞検知を再開する |

**使用例**: A/Bテストの結果を3日間待つ、キャンペーン効果を1週間観測するなど、外部の変化を計測するために意図的にループを止める場合に使う。

このフィールドは戦略層（LLMがタスクを生成するとき）が設定する。実行者が独自に変更することは禁止する。`plateau_until` が設定されているとき、`stall-detection.md` §2.5 の抑制ロジックが適用される。

### 2.7 見積もり所要時間

```
estimated_duration: Duration | null   // LLMがタスク生成時に見積もる所要時間
```

LLMがタスク生成時に推定する。単位は分または時間で表現する（例: `{ value: 3, unit: "hours" }`）。

`null` が許容される。見積もりが不可能なタスク（探索的調査、外部依存の多い作業）では `null` を設定する。

`null` の場合の停滞検知（`stall-detection.md` §2.2）の扱い: 時間超過チェックは絶対時間閾値（ゴールのドメインと過去の実績から算出）にフォールバックする。

### 2.8 連続失敗カウント

```
consecutive_failure_count: number     // 同種タスクの連続失敗回数（デフォルト: 0）
```

**これがシステム全体で唯一の失敗カウンターだ。** `stall-detection.md` の連続失敗検知と `curiosity.md` の繰り返し失敗発動条件は、ともにこのフィールドを参照する。

カウントの単位は「同一次元 × 同一タスクカテゴリ」の組み合わせ。まったく異なる戦略による別カテゴリのタスクが生成されたとき、カウントをリセットする。

**エスカレーション閾値: 3回**（設定可能パラメータ `failure_escalation_threshold`）。このしきい値を超えたとき、`task-lifecycle.md` §6 の `escalate` 判定が発動する。同じ数字が複数のドキュメントに散在することを防ぐため、ここで一元管理する。

> **注意**: `stall-detection.md` §4 の段階的対応（第1〜3検知）のステージカウンターは、停滞検知の**エスカレーション段階**を追跡するものであり、`consecutive_failure_count` とは別の概念である。前者は「何回目の停滞検知か」、後者は「同種タスクの連続失敗回数」を表す。両者はともに閾値3を持つが、リセット条件・スコープ・用途がそれぞれ独立している。

### 2.9 可逆性タグ

```
reversibility: "reversible" | "irreversible" | "unknown"
```

LLMがタスク生成時にタスクの可逆性を判定して付与する。

| 値 | 意味 |
|---|------|
| `reversible` | 失敗してもロールバック・取り消しが可能 |
| `irreversible` | 一度実行すると元に戻せない（削除、外部送信、デプロイ等） |
| `unknown` | 判定できない |

**`unknown` は `irreversible` と同等に扱う（保守的原則）。** 判定できない場合に「おそらく大丈夫」と楽観的に扱うことを禁止する。

このタグは実行前に `trust-and-safety.md` §4 の承認ゲートが参照する（詳細: `trust-and-safety.md` §4 参照）。`irreversible` または `unknown` と判定されたタスクは、トラストバランスや確信度スコアにかかわらず、常に人間の承認を必要とする。

## 3. スコープサイジング

タスクは「1回の実行セッションで完了できるサイズ」でなければならない。

### サイジング基準

**単一フォーカス**: 1つの次元、または密接に関連した次元群のみを対象とする。「フロントエンドのUIとバックエンドのAPIとインフラ設定を同時に」は単一フォーカスではない。

**検証可能な成功基準**: 実行後に「完了したかどうか」が客観的に判断できること。主観的な評価だけに頼る基準は、スコープが大きすぎるサインだ。

**限定された爆発半径**: タスクが失敗したとき、その影響が他の次元に波及しないこと。「このタスクが失敗しても、他の進捗は守られる」という設計。

### サイジングの失敗パターン

| パターン | 症状 | 対処 |
|---------|------|------|
| 過大 | 成功基準が3つ以上の独立した条件を含む | タスクを分割する |
| 過小 | 成功してもギャップがほとんど縮まらない | 関連タスクと統合を検討 |
| 曖昧 | 完了を判断するのにさらなる主観的判断が必要 | 成功基準を具体化する |

## 4. 実行

タスクは実行層（エージェント、API、人間など）に渡される。Conatusは実行中に**介入しない**。

### Conatusが渡すもの

実行開始時に、タスクのスコープ・成功基準・制約を含むコンテキストを組み立てて渡す（runtime.md §3 参照）。渡す情報は「このタスクを完了するために必要な最小限」に絞る。ゴール全体の文脈は不要だ。

### Conatusが監視すること

```
execution_state {
  status: "running" | "completed" | "timed_out" | "error"
  started_at: timestamp
  timeout_at: timestamp
  heartbeat_at: timestamp      // 最終応答確認時刻
}
```

Conatusが確認するのはこれだけだ。「進捗が見えない」「タイムアウトが近い」「異常終了した」の3種類の状態変化を検知し、対応する。実行の内容には介入しない。

### 実行中の不介入原則

実行者のコンテキストを汚染しないためだ。途中で「やっぱりこっちも直して」「ちなみにこの問題も」を差し込むと、スコープが崩れ、検証が不可能になる。指示は実行開始前に完結させる。

## 5. 検証（3層構造）

実行が完了した後、Conatusは3層の検証を行う。この構造は runtime.md §1「結果検証」に対応する。

### Layer 1: 機械的検証

```
mechanical_checks: Check[]    // 自動化された客観的確認
```

テスト実行結果、ファイルの存在確認、ビルドの成否、APIレスポンスのステータス。これが最も信頼度が高い。嘘をつけない種類の証拠だ。

機械的検証は**実行セッションとは別プロセス**で行う。実行者が「テスト通過」と報告しても、Conatusが独立して確認する。

#### Layer 1 の実行メカニズム

Layer 1 は、Conatusが専用の**検証セッション**を起動して実行する（実行セッションとは完全に別のセッション）。

検証セッションが受け取る情報:
- タスク定義（成功基準 + スコープ境界）
- 期待される成果物（ファイル、APIエンドポイント、ビルド成果物など）
- 実行セッションが生成した成果物へのアクセス手段

タスク種別ごとの検証内容:

| タスク種別 | Layer 1 で行うこと |
|------------|-------------------|
| コードタスク | テスト実行、型チェック、lint（エージェントセッション経由） |
| 非コードタスク | 期待ファイルの存在確認、フォーマット検証、APIレスポンスコード確認 |

**実行タイミング**: 実行セッションの完了直後、Layer 2 の前に実行する。

**Layer 1 が適用できない場合**: タスク種別に対して機械的な検証手段が存在しないとき、Layer 1 をスキップする。この場合、Layer 2 + Layer 3 のみで検証を行い、最終的な信頼度スコアを低下させた上で判定する。

信頼度: **高**

### Layer 2: タスクレビュアー

```
reviewer_session {
  input: {
    success_criteria,          // 成功基準（実行コンテキストなし）
    artifacts,                 // 成果物へのアクセス手段
    reference_standards        // 評価基準情報
  }
  output: {
    criteria_met: boolean[]    // 各成功基準の充足状況
    quality_assessment: string // 定性的評価
    concerns: string[]         // 懸念事項
    confidence: number         // 0.0〜1.0
  }
}
```

実行とは別のセッションで、独立したLLMが成果物を評価する。重要なのは、**実行セッションのコンテキストを渡さない**ことだ。実行時の文脈を共有しないからこそ、バイアスのない判断ができる。

信頼度: **中**

### Layer 3: 実行者の自己申告

```
executor_report {
  completed: boolean
  summary: string              // 何をしたか
  partial_results: string[]    // 完了した部分（未完了の場合）
  blockers: string[]           // できなかった理由
}
```

参考情報として扱う。自己申告だけで完了判断はしない。ただし、「何ができなかったか」「何が邪魔をしたか」は失敗対応の判断に使う。

信頼度: **低**

### Layer 1 と Layer 2 の矛盾解消ルール

Layer 1（機械的検証）と Layer 2（タスクレビュアー）が異なる判定を返したとき、以下のルールで解消する。

| Layer 1 | Layer 2 | 解消ルール |
|---------|---------|-----------|
| PASS | PASS | **PASS**（両者一致） |
| FAIL | FAIL | **FAIL**（両者一致） |
| PASS | FAIL | **要調査**: Layer 2 の懸念が意味的な問題（Layer 1 では検出できない種類）を示している可能性がある。追加コンテキストを付与した上で Layer 2 レビューセッションを再実行する。再実行後も FAIL なら **FAIL** として扱う。 |
| FAIL | PASS | **FAIL**: Layer 1（機械的・客観的）が優先する。Layer 1 の失敗は主観的評価では上書きできない。 |

**判断の根拠**: Layer 1 は「テストが通過したか」「ファイルが存在するか」という客観的事実を確認する。これは Layer 2 の「品質が十分か」という主観的評価よりも定義が明確だ。ただし、Layer 1 が通過しても Layer 2 が問題を指摘するとき、機械的には通過しながら意味的に欠陥がある可能性（型はあっているが振る舞いが間違っているなど）があるため、追加レビューで確認する。

### 検証結果の統合

3層の結果を統合して、タスクに信頼度スコアを付与する。このスコアが後続のギャップ認識（mechanism.md §2.2）に入力される。

```
verification_result {
  verdict: "pass" | "partial" | "fail"
  confidence: number           // 統合した信頼度
  evidence: Evidence[]         // 根拠となった観測
  dimension_updates: DimensionUpdate[]  // 各次元のスコア変化
}
```

## 6. 失敗対応の決定ロジック

タスクが完全には成功しなかった場合、Conatusは3つの選択肢から対応を選ぶ。

### 判断フロー

```
タスク失敗 / 部分的完了
    │
    ├─ 方向は合っているか？
    │       ↓ Yes（部分的な成果はゴールに向かっている）
    │       成果は再利用できるか？
    │             ↓ Yes → keep
    │             ↓ No  → 保持せず、方向を変えてdiscardへ
    │
    ├─ 方向が間違っているか？
    │       ↓ Yes（成果がゴールに向かっていない、または悪化させる）
    │       → discard
    │
    └─ 繰り返し失敗しているか？
            ↓ Yes（同種タスクが3回以上失敗）
            → escalate
```

### keep: 部分結果を保持して続行

適用条件:
- 部分的な成果がゴール方向に進んでいる
- 完了していない部分が明確に特定できる
- 残りの作業が別タスクとして定義できる

処理:
- 完了した部分の成果を状態に反映
- 未完了部分を新しいタスクとして生成
- 新タスクの先頭に「前回タスクの残作業」として文脈を付与

例: 10機能のうち7機能が実装済み → 7機能分のギャップ縮小を記録し、残り3機能を次タスクへ。

### discard: 成果を破棄して方向転換

適用条件:
- アプローチの方向性が間違っていた
- 成果がそのまま残ると他の次元を汚染・阻害する
- 同じアプローチで続けても改善の見込みがない

処理:
- 可能であれば変更をリバート（リバート可否の判定は下記「リバート可否の判定」参照）
- 失敗したアプローチの記録を次のタスク生成コンテキストに渡す（同じアプローチを繰り返さないため）
- 別のアプローチでタスクを再生成

例: データ構造の変更が既存機能を壊した → 変更をリバートし、影響範囲を調査するタスクを先に生成。

#### リバート可否の判定

discardパスに進んだとき、リバートを試みるかどうかはタスクの `reversibility` タグ（§2.9）を参照して決定する。不可逆アクションの定義は `trust-and-safety.md` §4 も合わせて参照すること。

| `reversibility` の値 | リバートの扱い |
|----------------------|--------------|
| `reversible` | 新しいエージェントセッションを起動し、リバートを試みる |
| `irreversible` | リバートを試行しない。次元の状態を `dirty`（汚染済み）としてマークし、人間にエスカレーションする |
| `unknown` | `irreversible` と同等に扱う（保守的原則） |

#### リバート失敗時の処理

`reversible` と判定されたタスクに対してリバートセッションを起動したが、リバート自体が失敗した場合:

1. **人間にエスカレーション**: 以下の情報をすべて含めて通知する。
   - 元のタスク定義と実行結果
   - 失敗した成果物の内容
   - 試みたリバートの内容と失敗理由

2. **次元状態の更新**: 対象次元に以下のフラグを設定する。

```
state_integrity: "uncertain"   // 手動観測が必要な状態
```

`state_integrity: "uncertain"` が設定された次元に対して、Conatusは次のタスクを自律的に選択・実行しない。人間が手動で状態を確認・更新し、`state_integrity` を `"ok"` に戻すまで、その次元は停止状態となる。

### escalate: 人間に介入を求める

適用条件:
- 同種のタスクが3回以上失敗している
- 必要な権限・能力をConatusが持っていない（本番環境のアクセス権、法的判断など）
- 前提が間違っている可能性があり、ゴール自体の再検討が必要な場合

処理:
- エスカレーション理由の記録（何を試み、なぜ失敗したか）
- 人間への通知（何を求めているかを明示）
- 待機状態に入る（他のゴールに注意を向けつつ、応答を待つ）

**判断の核心**: 「その作業はゴール方向に動いたか」。不完全でも方向が合っていればkeep。方向が間違っていればdiscard。自律的に解決できなければescalate。

## 7. タスク完了 → ループの継続

タスクが完了（または最終的に処理）されると、Conatusは即座に次のループを起動する。

```
タスク完了
    │
    ↓
状態ベクトル更新
  各次元の現在値を検証結果に基づいて更新
  観測信頼度も更新
    │
    ↓
ギャップ再計算
  更新された状態ベクトルを基に、全次元のギャップを再計算
    │
    ↓
ドライブスコア再計算
  締切駆動・不満駆動・機会駆動スコアを更新
  （機会スコアは時間経過でも減衰する）
    │
    ↓
次のタスク選択・生成へ
```

ループは止まらない。タスクが完了したことは、次のタスクを始めるトリガーだ。ゴールが達成されるか、ユーザーが止めるまで、このサイクルは続く。

---

## 設計原則のまとめ

| 原則 | 具体的な設計決定 |
|------|----------------|
| 優先順位はコードが決める | ドライブスコアによる次元選択はLLMに委ねない |
| 実行と検証の分離 | 検証は実行とは別セッション・別コンテキストで行う |
| スコープの明示的な境界 | `out_of_scope` の明記で「ついでに」を構造的に防ぐ |
| 失敗は情報 | 失敗したアプローチは記録して次の生成に渡す |
| ループは止まらない | タスク完了は終点ではなく、次のループの起点 |
