# Contributing to PulSeed

Thank you for contributing to PulSeed. This project focuses on reliable, evidence-based agent orchestration, so contributions should be clear, testable, and scoped to a specific improvement.

## How to Contribute

You can contribute by:

- Reporting bugs
- Proposing features or design improvements
- Improving documentation
- Adding or refining tests
- Submitting code changes

Recommended workflow:

1. Review the existing documentation in `README.md` and relevant source files before making changes.
2. Create a focused branch for your work.
3. Keep changes narrow in scope and avoid unrelated refactors.
4. Add or update tests when behavior changes.
5. Update documentation when commands, APIs, or user-facing behavior change.

## Development Setup

Use the following process to prepare a working local environment and verify it before you start changing code.

### Prerequisites

- Use Node.js 20 or newer. The repository declares `>=20` in `package.json`.
- Use the npm version that ships with your installed Node.js unless you have a project-specific reason not to.
- Confirm your environment before continuing:

```bash
node --version
npm --version
```

### Clone and Install

```bash
git clone https://github.com/my-name-is-yu/PulSeed.git
cd PulSeed
npm install
```

This project builds TypeScript into `dist/` and runs tests with Vitest. After installing dependencies, verify that the repository is healthy before you begin editing:

```bash
npm run build
npm test
npm run typecheck
```

If you are contributing from an existing clone, refresh your branch before starting work:

```bash
git checkout main
git pull --ff-only origin main
git checkout -b feat/your-change
```

### Local Development Workflow

Use the script that matches the part of the system you are editing:

- `npm run build`
  Recompiles the TypeScript source into `dist/`. Run this after changing anything in `src/` that affects runtime behavior.
- `npm test`
  Runs the full Vitest suite once. Use this before opening a pull request.
- `npm run test:watch`
  Starts Vitest in watch mode for fast iteration during implementation.
- `npm run typecheck`
  Verifies TypeScript correctness without writing build output.
- `npm run tui`
  Launches the terminal UI from `dist/tui/entry.js`. Build first so the output is current.

Example iteration loop for a change in the core loop or adapters:

```bash
npm run test:watch
# edit files in src/ and tests/
npm run typecheck
npm run build
```

Example workflow for a TUI change:

```bash
npm run build
npm run tui
```

Example workflow for a CLI change:

```bash
npm run build
node dist/cli-runner.js status
node dist/cli-runner.js report
```

### Provider Configuration

Many tests are deterministic and do not require real provider credentials. Only configure live provider access if your change genuinely needs it, such as:

- validating adapter behavior against a real API
- reproducing a bug in end-to-end CLI execution
- manually testing negotiation or verification prompts

OpenAI is the default provider. To switch providers, set `PULSEED_LLM_PROVIDER` explicitly.

```bash
export OPENAI_API_KEY=sk-...
export PULSEED_LLM_PROVIDER=openai
```

```bash
export ANTHROPIC_API_KEY=sk-ant-...
export PULSEED_LLM_PROVIDER=anthropic
```

When a manual test depends on live provider access, document that in your pull request so reviewers know what was and was not reproducible offline.

If provider-specific behavior is part of your change, note all of the following in your verification notes:

- the provider you used
- whether credentials were real or mocked
- the exact command or flow you exercised
- any non-default environment variables required to reproduce the result

### Repository Layout

Knowing the main directories will help you place changes and tests correctly:

- `src/` contains the application source, including adapters, context providers, shared types, and the terminal UI.
- `tests/` contains unit, integration, and end-to-end coverage. Keep new tests close to the behavior they validate.
- `dist/` contains compiled output generated by `npm run build`.
- `docs/` contains project documentation and design material.
- `pulseed-workspace/` is a local workspace used for project scenarios and examples.

Examples:

- If you change provider selection logic, update the corresponding test area such as `tests/provider-factory.test.ts`.
- If you change the CLI or run loop behavior, look at files such as `tests/cli-runner.test.ts` or `tests/core-loop-integration.test.ts`.
- If you change TUI behavior, add or update tests under `tests/tui/`.

### Local State and Reproducibility

PulSeed writes runtime state outside the repository, primarily under `~/.pulseed/`. That state can affect manual verification for persistence, reporting, and goal lifecycle behavior.

Before reporting test or manual-run results, be explicit about whether you used:

- an existing `~/.pulseed/` directory with prior goals and reports
- a cleaned state directory for a fresh run
- real provider credentials or mocked test execution

Example manual verification session:

```bash
rm -rf ~/.pulseed
npm run build
node dist/cli-runner.js goal add "Create a comprehensive README for this project"
node dist/cli-runner.js run
node dist/cli-runner.js report
```

Best practices:

- Start from a passing build, test, and typecheck when possible.
- Prefer deterministic tests over live-provider checks during normal development.
- Update `README.md` or adjacent docs in the same branch when you change setup commands, environment variables, or user-facing workflows.
- Do not commit files generated under `dist/` or state written under `~/.pulseed/`.

## Code of Conduct

All contributors are expected to participate professionally and respectfully.

- Be constructive in discussions, reviews, and issue reports.
- Assume good intent and focus on technical substance.
- Do not engage in harassment, discrimination, personal attacks, or disruptive behavior.
- Accept feedback and provide it in a clear, actionable form.

Project maintainers may moderate discussions and contributions to keep collaboration productive and safe.

## Submitting Issues

Open an issue when you find a bug, have a feature request, or want to propose a significant change.

Please include:

- A clear summary of the problem or proposal
- Relevant context and expected behavior
- Steps to reproduce, if reporting a bug
- Logs, error messages, screenshots, or test output when useful
- Notes about your environment, such as Node.js version or provider configuration, when relevant

Before opening a new issue, check whether the topic has already been reported.

## Pull Request Process

Pull requests should be small enough to review efficiently, specific enough to test directly, and documented well enough that reviewers can understand the behavior change without reconstructing your intent from the diff.

### Before You Open a PR

1. Create a focused branch from the latest `main`.
   - Good examples: `fix/reporting-null-state`, `feat/tui-goal-filter`, `docs/contributing-setup`.
2. Keep the branch scoped to one logical change.
   - Do not mix unrelated cleanup, opportunistic refactors, and feature work in the same pull request.
3. Update tests and documentation alongside the implementation.
   - Behavior changes should come with test updates.
   - User-facing command, setup, or output changes should come with docs updates.
4. Review the files you changed before pushing.
   - Look for generated output, local state artifacts, accidental formatting churn, and debug code.

### Required Local Checks

Run the standard verification sequence before requesting review:

```bash
npm run build
npm test
npm run typecheck
```

Add targeted verification when relevant. Examples:

- For TUI changes, include a short manual validation note after `npm run tui`.
- For CLI changes, include the exact command you ran and the observed behavior.
- For provider or adapter changes, specify whether validation used mocks or live credentials.
- For persistence changes, verify both a clean `~/.pulseed/` directory and an existing state directory when backward compatibility matters.
- For documentation-only PRs, confirm the examples still match the current scripts and command names in `package.json`.

### PR Description Template

Every pull request should answer these questions directly:

- What changed?
- Why was the change needed?
- How was it verified?
- What remains risky, manual, or intentionally deferred?

Use a structure close to the following:

```md
## Summary
- Adjust report generation when no active goal exists
- Add regression coverage for the empty-state branch

## Why
- `pulseed report` could fail on a fresh install before any goal was created

## Verification
- npm run build
- npm test
- npm run typecheck
- Manual: `node dist/cli-runner.js report` with a clean `~/.pulseed/`

## Notes
- Manual verification used a cleaned local state directory
- No live provider calls were required
```

For larger changes, add a short `## Risk` section describing migration concerns, state compatibility concerns, or areas you intentionally did not address in the same PR.

### Reviewer-Focused Expectations

Review your own diff before pushing and remove:

- debug logging
- commented-out experiments
- unrelated formatting churn
- accidental workspace or state-file changes

Be explicit about outputs that reviewers cannot infer from code alone. For example:

- paste a short CLI transcript if command output changed
- include screenshots if the TUI layout changed
- mention the test file names you added or updated
- note if a check was intentionally skipped and why
- mention whether the change affects `~/.pulseed/` persistence or provider configuration

### Updating During Review

After a pull request is open:

1. Respond to feedback with focused follow-up commits unless a maintainer requests squashing or a different workflow.
2. Re-run affected checks after each non-trivial code change.
3. Resolve review conversations only when both the implementation and explanation are actually complete.
4. Split out follow-up work if the scope grows materially during review.

Best practices:

- Optimize for reviewer comprehension over cleverness.
- State observable behavior changes, not vague intent.
- Call out risk around persistence, goal state, provider configuration, CLI output, and report generation when those areas are touched.



## Coding Standards

Follow these standards when contributing code or documentation so the codebase stays predictable, testable, and reviewable.

### General Expectations

1. Keep changes small and intentional.
   - Prefer focused updates over broad refactors.
   - Preserve existing module boundaries, naming, and file organization unless restructuring is part of the task.
2. Match the surrounding style.
   - Follow the conventions already present in the file for import order, naming, quoting, and test structure.
   - Do not introduce a new abstraction or helper pattern unless it solves a real repeated problem in this codebase.
3. Write code that makes behavior easy to inspect.
   - PulSeed is an orchestration system. State transitions, verification decisions, and provider selection logic should be obvious from the code path.
4. Prefer incremental changes to architecture-sensitive areas.
   - Core loop, persistence, verification, and provider dispatch changes should be introduced with minimal surface area and explicit tests.

### TypeScript Standards

- Keep the codebase passing `npm run typecheck`.
- Prefer explicit types at public boundaries and complex return values.
- Avoid unnecessary `any`, hidden mutation, and tightly coupled side effects.
- Favor clear data flow over clever indirection.
- Use descriptive discriminated unions or string literal types when representing orchestration outcomes such as statuses, verdicts, or task phases.
- Prefer existing shared types in `src/` over duplicating ad hoc object shapes in multiple files.

Example:

```ts
// Prefer explicit result shapes for orchestration decisions.
type VerificationResult = {
  verdict: "pass" | "partial" | "fail";
  confidence: number;
  rationale: string;
};
```

That style is preferable to loosely shaped objects that force reviewers to infer required fields from downstream usage.

Additional examples:

- Prefer `goalState.currentStatus` over generic names like `state.data`.
- Prefer a small helper that converts provider responses into a shared typed shape over repeating inline object assembly in multiple call sites.

### Testing Standards

- Treat behavior changes as test changes.
- Add or update Vitest coverage when modifying logic in `src/`.
- Place tests in the most relevant existing area under `tests/`.
- Prefer deterministic tests over network-dependent checks whenever possible.
- When fixing a regression, write the test so the failure mode is obvious from the test name and setup.
- Keep fixtures and setup local to the test file unless they are genuinely reused across multiple suites.

Examples:

- A change to provider dispatch should update coverage near `tests/provider-factory.test.ts`.
- A change to persistence or reports should include regression coverage near `tests/state-manager.test.ts` or `tests/reporting-engine.test.ts`.
- A change to TUI interactions should update tests under `tests/tui/`.

When you fix a bug, add a test that fails before the fix and passes after it if the behavior can be automated.

### CLI, Persistence, and External Effects

Be deliberate when modifying code that affects:

- files written under `~/.pulseed/`
- CLI command behavior or output
- task execution and verification decisions
- provider credentials, selection, or network calls

Changes in these areas should:

- preserve backward compatibility where practical
- make failure modes explicit
- avoid silent format changes to persisted state
- be described clearly in the pull request with reproduction steps
- keep command examples in documentation aligned with the actual built entry point `node dist/cli-runner.js` or the packaged `pulseed` binary

If you change user-visible output, include an example in tests or in the pull request description.

### Documentation and Comments

- Update docs when commands, workflows, environment variables, or user-facing behavior change.
- Keep inline comments brief and focused on intent, not restating obvious code.
- Remove temporary debugging artifacts before submission.
- Prefer concrete examples over abstract instructions when documenting CLI flows or environment setup.

### Practical Style Rules

- Prefer descriptive names such as `verificationResult` or `goalState` over ambiguous names like `data` or `result`.
- Keep functions focused on one orchestration concern where possible.
- Reuse existing helpers before adding new utilities.
- Avoid drive-by formatting changes in files unrelated to your task.
- Preserve the surrounding import and export style in the file you are editing instead of normalizing unrelated code.
- Keep examples and snippets in Markdown synchronized with the commands contributors can actually run in this repository.

Best practices for verification:

- Use targeted tests during development, then run the full build, test, and typecheck sequence before opening a pull request.
- If manual or live-provider validation is required, document exactly what you ran.
- If a bug depends on prior state, specify the setup clearly in the test or PR description.
